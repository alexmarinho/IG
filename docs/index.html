Story · why order matters

Order changes time. Time changes cost.

A best order does exist — but there are more of them than you could ever check. So the game isn't to try them all; it's to guess well and improve fast.

The catch

So why not just try every order?

Because there are too many. Drag the number of jobs and watch the count of possible orders run away from you: even at a billion plans checked every second, trying them all outlasts the age of the universe long before you reach a real 28-job problem. That is why you have to guess — and guess well.

The method

So you guess — then destroy, rebuild, and keep what's better.

A real Iterated Greedy engine running in this page on a seeded 12-job, 4-family instance (mulberry32, seed 0xce29767d). Nothing is pre-recorded: every insertion position is re-evaluated against the complete objective while you watch. Watch for the five beats — construct by best insertion, destroy three jobs, measure every insertion on the trial track, reject at break-even when no position beats the rejection cost, accept or revert.

§6Technique lab — same animation, three renderers

The demo-1 Gantt, implemented three times with three different technologies — genuinely separate implementations, shuffled in sync, so the code styles can be compared line by line. Open the source and compare labDom.render, labSvg.render, labCanvas.render.

one order, three renderers

(a) DOM + CSS transitions

<div> segments · animate style.left / style.width via CSS transition

Strengths: accessible by default, easiest layout & text, FLIP-friendly, trivially inspectable. Limits: comfortable to ~hundreds of nodes; animating layout properties (left/width) costs style recalc — fine for 5 segments, wrong for 5,000.

(b) SVG + JS attribute animation

<rect> nodes · rAF tween writes x / width attributes

Strengths: crisp at any zoom, still a DOM (inspectable, styleable), the natural fit for diagrams & timelines. Limits: heavier per node than canvas; no layout engine — you compute every x yourself.

(c) Canvas 2D redraw loop

single <canvas> · full redraw every frame

Strengths: fastest for thousands of marks, zero DOM overhead, pixel-exact. Limits: manual hit-testing, no accessibility tree, redraw-everything mindset; text & DPI are your problem.

Verdict

§8Recommended stack — three concrete paths

Path 1 — your case

Interactive explorable single-file page

Vanilla JS + WAAPI/FLIP, exactly as this mockup: one central clock, FLIP chip choreography, a live trial track. Add GSAP 3.15 + Flip/ScrollTrigger as a drop-in accelerator and D3 for the factorial counters and convergence charts. MIT-pure alternative: Anime.js v4.

  • Why: zero build step preserved — the page stays one file; FLIP is precisely what “remove / re-insert a job” needs.
  • Tradeoffs: hand-rolled timing demands discipline (single clock, seeded RNG); GSAP is free incl. commercial but “Standard No-Charge”, not MIT.

Migration effort: days. Keep your engine and DOM; port one beat at a time — this page is the proof of feasibility.

Path 2 — long-form

Rendered narrative video

Manim CE 0.20 for math fidelity (Python, 3Blue1Brown-grade exposition), or Motion Canvas 3.17 if you want to stay in TypeScript with generator-function storyboards and a visual editor.

  • Why: frame-perfect control, voiceover sync, YouTube-native; LLM-assisted Manim authoring is now a mainstream workflow.
  • Tradeoffs: loses interactivity — a second artifact to maintain; Motion Canvas public development has been quiet since late 2024.

Migration effort: 1–2 weeks. The storyboard mirrors the beats in Fig. 03; your seeded instance and objective port directly.

Path 3 — distribution

Clips & polish

Remotion 4 for social/marketing cuts (free for individuals & orgs ≤ 3 people; renders locally, on Lambda, or in-browser via WebCodecs), Rive for interactive mascots/operators with Data Binding wired to live algorithm state, dotLottie for exported micro-animations.

  • Why: distribution formats reward polish; Rive state machines make a reactive “operator” character cheap.
  • Tradeoffs: three extra toolchains — keep them out of the core explainer; Remotion’s license turns paid above 3 employees.

Migration effort: incremental — nothing until you need distribution; clips reuse the same beats and palette.
